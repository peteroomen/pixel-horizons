## Goal

Every combat action lands with weight: playing a card punches, damage floats off the
target, the struck plate flashes/shakes, AP and shield pips pop on change, and big hits
shake the viewport — all derived from the existing per-event `CombatView` snapshots, with
the sim and `CombatView` shape left **unchanged**. Respects `prefers-reduced-motion`.

## Approach

**Core technique — fx from snapshot deltas, not from the sim.** Combat already pushes a
fresh `CombatView` to React once per event (`onCombatUpdate`, never per-frame — ADR + the
"no React state at 60fps" rule). A new `useCombatFx(view)` hook keeps the *previous* view
and diffs it against the incoming one to emit discrete fx triggers: enemy/organ HP drop →
floating damage + plate flash/shake; ship HP drop → ship flash/shake; AP change → pip pop;
shield layer lost/recharged → pip pop. This keeps `src/game/sim/` and `combat-view.ts`
**untouched** (sim stays sacred; zero overlap with the shield branch's `combat.ts` edits)
and means all juice is presentation-layer.

- If a delta is genuinely ambiguous (e.g. damage to a specific organ vs. core in the same
  snapshot), prefer per-part diffing over threading an fx event through the sim. Only if
  that proves impossible do we add a **read-only, optional** field to `CombatView` — avoid
  if at all possible, and never add sim logic.
- **DOM for cards and floating numbers** (ADR 001 — cards are DOM; the enemy/player plates
  are DOM too). Floating damage numbers render as a positioned DOM overlay anchored to the
  plate that took the hit. **Pixi (`space-renderer.ts`)** owns the ship-sprite hit flash and
  the viewport shake.
- **Motion** (already a dependency, currently unused — CLAUDE.md known issue) drives the
  DOM card animations: hover-lift, play recoil/punch, fly-to-discard, draw deal-in. No new
  library, no ADR.
- **`prefers-reduced-motion`**: a tiny `useReducedMotion` gate (Motion ships one) disables
  shake and large translations, keeping only opacity/color cues. Accessibility + avoids
  motion sickness.

## Gotchas

- **Keep the sim sacred:** no edits under `src/game/sim/` or `surface/`; `combat-view.ts`
  ideally untouched. All juice is renderer/UI. This is also what keeps the merge with the
  shield branch clean.
- **No React state at 60fps:** animations run via Motion/CSS keyframes off discrete
  view-change events, not a per-frame state loop. The fx hook fires on snapshot change only.
- **Integer zoom is device-pixel based** (`renderer/pixel-scale.ts`) — viewport shake must
  not fight the integer-scale letterbox; shake the inner content/stage, not the scaling math.
- **`touch-none` stays on the canvas host**, not on DOM overlays — the floating-number layer
  must be `pointer-events: none` so it never eats card taps.
- Node 22 before install/dev/commit; lint is `eslint src`; `docs/` is `.prettierignore`d.
- Pure renderer/UI PR — does **not** violate "don't mix sim + renderer in one PR" (no sim).

## What actually happened

Built exactly to plan — fx derived from `CombatView` deltas, sim and `CombatView` shape
untouched. Key decisions confirmed in the build:

- **Co-located per-element hooks** rather than a global fx layer. `useDamageFloaters(hp)`
  lives on each plate / organ / hull, so a floating number always lands on the exact bar
  that took the damage with **no global coordinate math**. Organ rows were extracted into
  an `OrganRow` component so each can own its own hook (hooks can't run in a `.map`).
- **Pure core split out** (`combat-fx-core.ts`, React/DOM/Pixi-free) so the diff math is
  unit-testable under vitest's `node` env (there's no jsdom — component/hook tests aren't
  possible, so the tested surface is the pure `hpDrop` + `shakeAmplitude`).
- **Renderer shake** lives in `space-renderer.ts`, driven off the HP deltas it already
  diffs in `sync()` — no new GameHandle/React→Pixi plumbing. The backdrop is **overscanned
  by `ceil(MAX_SHAKE/PX)` tiles** and offset so a shake never slides the void into frame.
  Shake is a **damped sinusoid, not RNG** (RNG would consume the sim's deterministic
  stream), integer px for crisp nearest-neighbour, and **gated by `prefers-reduced-motion`**
  via `matchMedia` at renderer creation.
- **Card identity:** the view's card key embeds the hand index, which shifts the whole tail
  when a card leaves. Switched the hand's `AnimatePresence` keys to **id + occurrence
  ordinal**, so only same-id duplicates after the removed card shift — most cards keep
  identity and only the played card animates out. `mode="popLayout"` + `initial={false}`
  gives play-out + draw-deal-in without a burst when combat opens.
- `EnemyPartView` had to be re-exported from `main.ts` (it wasn't).

**Browser-verified** (dev server, Gatemaw boss via `?enemy=enemy-gatemaw`): played Cannon
Burst → enemy core floater rendered `−3` and core HP went 70→67 (value-accurate); shield
layers spent → pip recharge counters render via `PopPip`; infested/normal card states
render through the Motion wrappers; **no console errors**; layout intact at desktop **and
375px**; backdrop shows no void edge from the shake overscan. Transient flash/shake/rise
animations were confirmed by catching the floater element in the DOM right after a hit
(screenshots can't catch a <750 ms animation across a tool round-trip). 497 tests green
(7 new), lint + type-check + `pnpm build` clean.
