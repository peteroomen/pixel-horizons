## TL;DR

Slice A shipped a working but **incrementally-good** version of the two lane
transitions. The structure is right (ship in frame, pixelated coral spiral, camera
push, planet-rise + seamless orbit handoff on the Drop, engine-wash spool-up on the
Launch), but the **feel is not final** — the spiral reads a bit sparse/scattered
and the smoothness/timing wants a proper tuning pass with human eyes. Two things
remain: **(1) a feel/design polish pass** on Drop+Launch, and **(2) Slice B — Drop
Pod Deploy** (orbit → surface), built on the same driver.

## How to see it (the key tool)

`?transition=lane-drop` or `?transition=lane-launch` loops the transition in
isolation — the only sane way to tune feel. Combine with `?hull=hull-scout` /
`?modules=…` to preview the in-frame ship. (Added this session; in the dev-knobs
table in `CLAUDE.md`.)

⚠️ **Gotcha:** the headless preview tool throttles `requestAnimationFrame` between
screenshots, so the animation barely advances frame-to-frame in screenshots — you
**cannot** judge smoothness or catch the mid-/late-animation phases (planet-rise,
flash beat, seamless settle) from headless screenshots. This pass needs a **real
foreground browser**. The composition (ship present, pixelation, hull, engine wash)
is all that was verifiable headlessly, and it checks out.

## Where everything lives

- **Driver:** `src/renderer/transition.ts` — `playTransition(app, kind, assets, onComplete)`.
  One timeline, `kind: 'lane-drop' | 'lane-launch'`, reversible via the speed curves.
- **Seams:** `src/game/main.ts` — `runTransition(kind, assets, onComplete)` sets the
  `'transition'` phase (canvas-only, no DOM) then plays the overlay. Lane Drop wraps
  the planet branch of `arriveAtNode` → `enterOrbit`; Lane Launch wraps `selectNode`
  → `startLane`. `activeTransition` is cancelled in the handle `destroy()`.
- **Ship asset:** `compositeShipFromModules(hullId, names)` in `src/renderer/sprites.ts`
  (the shared composite combat + orbit use). `shipCanvas()` helper in `main.ts`.
- **Phase:** `'transition'` in the `GamePhase` union; added to the world-framing
  condition in `src/app/page.tsx` (the Pixi canvas is always mounted, page.tsx ~180).
- **Design source of truth:** `docs/design/foundry-transitions-ships.dc.html` — live
  canvases + keyframe storyboards + timing for all three transitions (Drop, Launch,
  Pod Deploy). Re-read this; the FX-beat law (five accents) is in there.

## Tuning levers (all constants at the top of `transition.ts`)

- `CHUNK` (7) — pixel-square size of the spiral. Bigger = chunkier/more pixelated.
- `SPIRAL_POINTS` (150), `D_THETA` (0.22) — density and tightness of the spiral arm.
  **The sparse look is here** — try more points and/or a second interleaved arm for a
  denser tunnel, or thicker squares.
- `REACH` — how far the spiral fills the frame.
- Speed curves inside `tick()` (`isDrop` branch vs launch branch) — the decel/accel
  shape. The Drop decelerates to ~0 by t=0.7 then flashes; the Launch spools to t=0.24
  then accelerates. Easing helpers (`easeOutCubic`/`easeInCubic`/`easeInOut`) are there.
- `cam` lerps — camera push amount (Drop 1.12→1.0, Launch 0.96→1.12).
- Drop settle window `(t-0.62)/0.38` — when the ship eases to `ORBIT_SHIP_X/Y` and the
  planet rises to centre. **Must stay matched to `orbit-renderer.ts`** poses
  (`SHIP_X=128`, `SHIP_Y=H-86`, planet centre, `PLANET_SIZE=200`) or the handoff jumps.
- `DROP_MS` (2400) / `LAUNCH_MS` (2500) / `REDUCED_MS` (260) durations.

## Known gaps / open questions for the design pass

1. **Spiral density/feel** — currently reads scattered, not a continuous "tunnel."
   Densify, or reconsider the shape (the design doc shows scaling near/far coral
   *wall strips* + speed-lines, not necessarily a spiral — the user asked for a
   spiral, so reconcile the two).
2. **Seamless orbit handoff** — coded (ship + planet settle to the orbit pose under
   the flash) but **unverified in motion**. Check the Drop→orbit cut has no jump.
3. **Smoothness** — the whole point; needs foreground 60fps eyes.
4. **Lane Drop is only reachable in real play via a planet arrival** (win a lane to a
   planet node). Use `?transition=lane-drop` to iterate; verify the real seam once.
5. **Reduced-motion** path is a bare flash — fine, but confirm it reads.

## Slice B — Drop Pod Deploy (orbit → surface)

Built on this same driver. Design (doc §3, ~3.6s): Scene A holds on the ship as the
pod detaches and shrinks toward the planet; at atmosphere-contact an amber heat-glow
peaks and **hides the cut**; Scene B rides the pod down — streak in, retro-burn,
1-frame red impact flash + 3px shake, ramp drops, clone deploys. Seam: `dropToSurface`
→ `enterSurface` in `main.ts`. Reuse the overworld pod sprite (§2) at 3 scales; the
planet sprite is the same orbital one. Return trip (surface→orbit) is a plain fade —
no bespoke animation. A `?transition=pod-deploy` knob would help (extend the existing
dev knob + `TransitionKind`).
