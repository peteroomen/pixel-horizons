# Station UX & Workbench Access

**Date:** 2026-06-15
**Branch:** claude/sharp-wright-iv37l2
**Roadmap item:** Phase 4 — 4.8 Station UX & workbench access

## Goal

Make module management **legible** and **reachable**: open the Workbench from
Merchant/Engineer nodes, show the full slot picture (used / free / required),
preview a module's generated cards before buying or installing, and explain
keywords/mechanics with the 4.7 `InfoChip` tooltip — all readable at 375px.

## Approach

A view-layer + UI slice (no Pixi renderer changes). Rules stay in the sim; the
UI only *explains* them.

- **Slot picture (sim → view):** expose the already-private slot math in
  `sim/economy.ts` as `slotUsage(hullId, modules)` → `{ slot, used, limit }[]`.
  Project it into `ShipView.slots` (`ship-view.ts`) and `MerchantView.slots`
  (`station-view.ts`). The component renders it; it never recomputes the rule.
- **Generalize the buy-block reason:** replace the `'need-slot' | 'need-scrap'`
  string on `ShopOfferView` with a structured `OfferBlock` carrying the full
  picture — `{ kind: 'need-slot', slot, used, limit }` /
  `{ kind: 'need-scrap', price, have }` — so a blocked buy says *which* slot is
  full (X/Y) or how much Scrap is short.
- **Card preview (view layer):** add `describeModuleCards(moduleId, tier)` to
  `combat-view.ts`, reusing the existing private `describeCardText` +
  `cardKeywords`. Bake the result into each module/cargo/offer entry so the
  component renders handed-down data (respects the React↔sim boundary). Render
  as tap-to-expand to keep 375px density manageable.
- **Workbench reachable from stations:** `installModule`/`uninstallModule`
  already accept `shop`/`engineer` phases in `main.ts`; relax the `openWorkbench`
  guard to those phases too and mount the `Workbench` overlay over the station
  screens in `page.tsx` via a WORKBENCH affordance on `StationScreen`.
- **Tooltips:** wrap the station/workbench surface in `<InfoChipProvider>` and
  reuse `InfoChip` + `KEYWORD_GLOSSARY` (and `getStatus` where a card applies a
  status) so explanations match combat.

## What actually happened

Built as planned — a pure view/UI slice, no Pixi renderer changes.

- **Sim:** added `slotUsage(hullId, modules)` + `SlotUsage` to `economy.ts`, exposing
  the previously-private per-slot occupancy as a clean projection (rule stays in the sim).
- **View layer:** `combat-view.ts` now exports `describeModuleCards(moduleId, tier)` +
  `ModuleCardView`, reusing the existing private `describeCardText`/`cardKeywords` so a
  module's deck contribution renders identically to combat. `ship-view.ts` carries
  `slots` + per-module/cargo `cards`. `station-view.ts` replaces the
  `'need-slot' | 'need-scrap'` string with a structured `OfferBlock` (slot name + used/limit,
  or price + have) and adds `slot`, `cards` per offer plus `slots` on the merchant view.
- **Entry point:** `openWorkbench` now accepts `shop`/`engineer` (install/uninstall
  already did); `StationScreen` gained a WORKBENCH button beside Leave, and `page.tsx`
  mounts the `Workbench` overlay over the station phases. Workbench-open now resets on
  every phase change (not just map).
- **UI:** new shared `slot-labels.ts` + `foundry/ModuleCardList.tsx` (tap-to-reveal card
  previews with keyword `InfoChip`s). Workbench shows per-slot used/limit headers and
  expandable module cards; the merchant shows a slot-picture summary, per-offer slot +
  card preview, and the full block-reason copy. Both surfaces wrapped in `InfoChipProvider`.
- `ModuleSlot` + the new view types re-exported from `@/game/main` so components keep
  importing only through `main` (boundary preserved).

Tests: 441 green (added `slotUsage`, `station-view`, `ship-view` suites). Lint, type-check,
and `pnpm build` all clean. **Browser verification at 375px is still a human step.**
