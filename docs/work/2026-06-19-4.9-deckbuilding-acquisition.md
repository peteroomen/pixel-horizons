# Deckbuilding Acquisition & Starting-Deck Audit

**Date:** 2026-06-19
**Branch:** feature/deckbuilding-acquisition
**Roadmap item:** Phase 4 — 4.9 Deckbuilding acquisition & starting-deck audit

## Goal

Make the deck visibly grow across a run. Two confirmed changes from the
2026-06-15 playtest: (a) start with fewer installed modules so each acquisition
is a bigger relative change to the deck and ship silhouette; (b) make more
modules flow as mid-run rewards (events + map-node pacing + shop offer count) so
the deck you reach the gate with is one you assembled. Absorbs the A5
starting-loadout audit deferred from 4.6.

## Approach

Pure-data tuning + test churn — no sim/renderer logic changes, no shop-stock
logic (that's 4.13's lane). Keeps every change inside `data/` + tests so it runs
in parallel with 4.13 without touching a shared file.

1. **Trim starting modules** (`data/hulls.ts`). Drop one combat-slot module per
   hull so each starts sparser. Every hull keeps `mod-shield-generator` (implicit
   shield slot) and `mod-standard-print-matrix` (clone-bay). Proposed trims —
   **flag for confirmation, these are balance calls:**
   - **Gunship** 3W/1U/1E: drop `mod-autocannon` → starts Flak + Missile (2 of 3
     weapon slots filled; the 3rd weapon becomes a meaningful early pickup).
   - **Scout** 1W/2U/2E: drop `mod-phase-shifter` → starts Light Laser + 2×
     Thruster (both engines keep the dart identity; a utility slot opens up).
   - **Freighter** 1W/1U/2E: drop `mod-cargo-scanner` → starts Mining Laser +
     Hauler Engine (utility slot opens; salvage identity intact).
   - **Tactical** 2W/2U/1E: drop `mod-missile-pod` → starts Light Laser + Cargo
     Scanner + Thruster.
   This **shifts module indices** and will churn tuned combat tests that build
   decks from hull defaults — expected, take the churn (this is why A5 was
   deferred). Tests that construct explicit module lists are unaffected.

2. **More module-granting events** (`data/events.ts`). The `gain-module-to-cargo`
   outcome already exists (Derelict Freighter). Add 2–3 events whose primary draw
   is a module pickup (a cache/wreck/abandoned-station flavor), so events become a
   reliable module-injection point, not just resources/modifiers.

3. **Map node pacing** (`data/constants.ts`). Rebalance `MAP_NODE_WEIGHTS` so
   acquisition nodes (shop + event) appear more often across the 4 middle columns —
   e.g. nudge `shop`/`event` up relative to `cache`. Keep the planet floor intact.

4. **Shop offer count** (`data/economy.ts`). Bump `SHOP_OFFER_COUNT` (3 → 4) and
   optionally widen `SHOP_MODULE_POOL` so a merchant visit presents more choice.
   This is a *constant value* edit only — the shop-stock **logic** in
   `shop-inventory.ts`/`station-view.ts` is 4.13's; this slice does not touch it.

## What actually happened

Pure-data slice as planned, executed inside a parallel-safe worktree alongside
4.13. All four data edits landed exactly as confirmed:

- **Hull trims** (`data/hulls.ts`): Gunship dropped `mod-autocannon` (now
  Flak + Missile in 2 of 3 weapon slots); Scout dropped `mod-phase-shifter` (a
  utility slot opens); Freighter dropped `mod-cargo-scanner` (a utility slot
  opens); Tactical dropped `mod-missile-pod`. Every hull keeps its Shield
  Generator + Standard Print Matrix.
- **Three new module-granting events** (`data/events.ts`) modelled on Derelict
  Freighter, each a module-pickup vs. resources fork: `event-weapons-cache`
  (Kinetic Railgun / 6 hull cost), `event-drifting-prospector` (Mining Laser /
  4 scrap), `event-abandoned-relay` (Phase Shifter / free). With Derelict
  Freighter that's now 4 of 6 events that can inject a module.
- **Map pacing** (`data/constants.ts`): `MAP_NODE_WEIGHTS` shop 2→3, event 2→3
  (planet floor of 4 and cache 1 untouched), so acquisition nodes surface more
  often across the 4 middle columns.
- **Shop offers** (`data/economy.ts`): `SHOP_OFFER_COUNT` 3→4 (constant value
  only; shop-stock *logic* stayed untouched — that's 4.13's lane).

Test churn was exactly the index/loadout shift A5 predicted, fixed in five test
files. The two non-obvious ones: (1) `combat.test.ts` "malfunctioning instance
never retains" started failing not from an index shift but because the trimmed
Gunship draw pile is now small enough that the post-discard draw reshuffled the
just-discarded card back into hand — fixed by stocking the draw pile so the
intent (a malfunctioning Retain card discards) is tested without the reshuffle
artifact; (2) the `economy.ts` upgrade/uninstall tests looked up
`mod-phase-shifter` in the default Scout loadout, which no longer carries one —
fixed by pushing a phase-shifter into the run before the lookup (Scout has a
free utility slot now), preserving each test's intent.

**Deviations from the plan:**
- `catalog.test.ts` needed **no** edit: its starting-module assertions use `<=`
  slot bounds and only pin shield + matrix presence, so the trims pass as-is.
- `map-gen.test.ts` needed **no** edit: it asserts structural invariants and a
  planet floor, not exact weight-derived counts, so the reweighting passes.
- Plan said "optionally widen `SHOP_MODULE_POOL`" — left the pool as-is; 4 offers
  from the existing 11-module pool keeps offers distinct and the choice wider.
