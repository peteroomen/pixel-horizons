# Handoff → 6.1 slice 2: Surface-tileset recolor (the "biomes match the planet" payoff)

**Date:** 2026-06-18
**For:** the next session
**Assumes:** you start cold. Slice 1 (runtime planets + orbit screen) is merged to main.

## TL;DR

The planet you see from orbit is generated from a Resurrect 64 **ramp** (`planetRamps(seed)`).
The surface tileset is already built to be **recoloured by swapping that ramp**. Slice 2 wires
the two together: the surface you drop onto is tinted to match the planet you just saw — ADR
002 §3 / the whole reason runtime generation was worth it. Coordinate with **5.3 (Biomes 2–3)**.

## Why this is mostly plumbing, not new tech

Both halves already exist and were built to meet here:

- **Planet side** (`src/renderer/palette.ts`, merged): `planetRamps(seed): { ocean: Ramp; land: Ramp }`
  returns 6-step ramps (index 0 = lightest .. 5 = darkest).
- **Surface side** (`src/renderer/surface-sprites.ts`, from PR #20): every terrain tile factory
  takes a ramp param, e.g. `rockFillTile(P: Ramp = ROCKY, seed)`, `rockSurfaceTile(P, seed)`.
  The file's own comment: *"One ordered ramp per material so the whole tileset LUT-swaps as a
  unit later (Volcanic/Ice are the same ramp positions, different hues — wired in 5.3)."*
  `ROCKY`/`VOLCANIC`/`ICE` are exported 6-step ramps in the **same format** `planetRamps` emits.

So the job is: pass the planet's `land` ramp (or a per-biome ramp derived from it) into the
surface renderer's tile creation instead of the hardcoded `ROCKY`.

## The wiring path (where to cut in)

1. **Get the descriptor to the surface.** Planet identity is `planetForNode(run.seed, nodeId)`
   (`src/game/sim/planet.ts`) — already used by `enterOrbit` in `src/game/main.ts`. The surface
   must use the **same** descriptor so orbit and ground agree. Cleanest: in `main.ts`, compute
   the descriptor once on planet arrival and pass it through `enterSurface` →
   `startSurfaceMode` (`src/game/modes/surface-mode.ts`) → `createSurfaceRenderer`
   (`src/renderer/surface-renderer.ts`). Or recompute `planetForNode(run.seed, run.position.nodeId)`
   inside `enterSurface` (deterministic, same result) — but mind the dev knobs (below).
2. **Map descriptor → surface ramp.** For terran, `planetRamps(descriptor.seed).land` is the
   terrain ramp. (5.3 will introduce Volcanic/Ice planet *types*; for now terran → its land ramp.
   Consider a small `surfaceRampFor(descriptor)` helper next to `planetRamps`.)
3. **Feed the renderer.** `surface-renderer.ts` currently builds tiles with the default ramp;
   thread the chosen ramp into the `*Tile(P, …)` calls. Keep `palette.ts` the single source of
   the ramp; `surface-sprites.ts` stays a pure factory.

## Gotchas / decisions to make

- **Dev knobs.** `?mode=surface` drops straight to the surface with **no node/orbit** — it has
  no descriptor. Give it a default (e.g. `planetForNode(seed, 'dev-surface')`). `?mode=orbit`
  arrives via `enterOrbit('dev-orbit')` then DROP → `enterSurface`; make sure DROP carries the
  same descriptor so the dev orbit planet and dev surface match.
- **Determinism unchanged.** Descriptor is derived, not stored — don't add it to `RunState`
  (no `RUN_STATE_VERSION` bump). Same seed+node ⇒ same planet ⇒ same surface tint.
- **"Two palettes, one law."** PR #20's rule: terrain is muted R64 ramps; pure FOUNDRY accents
  are reserved for signals (the core-crystal cyan glint is the one saturated terrain pixel).
  Don't let a recolor push terrain into pure-accent territory — `planetRamps`' ramps are
  already muted R64, so this holds, but keep it in mind if you derive new ramps.
- **Readability floor.** A very low-contrast planet ramp could make terrain hard to read
  (platforms vs background). `palette.ts` has `snapToPalette`; consider a contrast guard on the
  surface ramp (the plan flagged "degenerate ramp still renders a readable surface" as a test).
- **Non-terrain tiles.** Bedrock, hazards (`brambleTile` uses `ROCKY[4]`), pod, items — decide
  which recolor with the biome and which stay fixed. Probably terrain + crust recolor; signals/
  props stay constant.
